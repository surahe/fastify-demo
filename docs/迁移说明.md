# 项目结构迁移说明

## 文档目的

本文档用于说明：如果你准备把当前这个 `fastify-demo` 的工程结构复制到真实项目中，哪些文件可以直接复用，哪些文件需要改名，哪些文件只适合保留思路而不适合原样照搬。

本文档只讨论工程结构迁移，不讨论真实业务实现细节。

## 一、迁移策略

建议采用下面这套方式迁移：

1. 先复制当前项目的工程骨架。
2. 保留通用工程能力，例如配置、错误处理、插件装配、日志、限流、监控、测试脚本。
3. 删除或重写 demo 业务模块。
4. 按真实项目模块重新建立 `routes / schema / controller / service / types` 结构。
5. 先实现一条最关键的业务主链路，再逐步补其他模块。

建议把当前仓库当成：

```text
可迁移的工程骨架
+ 少量学习型演示业务
```

不要把它当成可以直接上线的业务代码模板。

## 二、可以直接原样复制的文件

下面这些文件或目录，适合直接复制到新项目：

### 2.1 应用装配与启动

- `app.ts`
- `server.ts`

这两份文件负责：

- 创建 Fastify 实例
- 注册 core plugins
- 自动加载 route plugins
- 统一错误处理
- 统一 404 返回
- 启动服务

这部分设计和业务本身耦合较低，适合直接保留。

### 2.2 配置与环境变量

- `config/index.ts`
- `.env.example`

其中：

- `config/index.ts` 的组织方式适合保留
- 真实项目里建议继续按配置域拆分 schema，例如 `server / fastify / auth / upstream`，再合并成统一配置入口
- `.env.example` 建议复制后立刻按真实项目调整字段

### 2.3 通用错误处理

- `errors/app-error.ts`
- `errors/error-codes.ts`

这两份文件可以作为新项目统一错误体系的基础版本。

### 2.4 通用基础能力

- `lib/cache/ttl-cache.ts`
- `lib/http/http-client.ts`
- `lib/http/upstream-registry.ts`
- `lib/observability/metrics-store.ts`

这些文件本质上是平台能力，不是 demo 专属逻辑，适合继续保留。

### 2.5 Fastify 类型扩展

- `types/fastify.d.ts`
- `types/request-context.d.ts`

这部分适合直接迁移，因为真实项目也会需要 Fastify 实例扩展和 request context 类型补充。

### 2.6 Core Plugins

- `plugins/core/request-context.ts`
- `plugins/core/security.ts`
- `plugins/core/cors.ts`
- `plugins/core/platform.ts`
- `plugins/core/observability.ts`
- `plugins/core/rate-limit.ts`
- `plugins/core/docs.ts`
- `plugins/core/under-pressure.ts`

这些插件是整个项目的“平台层骨架”，非常适合继续沿用。

### 2.7 平台路由

- `plugins/routes/health.ts`
- `plugins/routes/metrics.ts`

这两类接口通常在真实项目里也长期存在，建议保留。

### 2.8 工程配置

- `package.json`
- `tsconfig.json`
- `eslint.config.mjs`
- `.prettierrc.json`
- `.prettierignore`
- `.editorconfig`
- `.github/workflows/ci.yml`

这些文件建议整体复制，再按新项目名称、依赖和脚本做少量调整。

## 三、复制后建议立刻修改的文件

下面这些文件虽然适合复制，但复制后建议第一时间改掉其中的项目元信息或文案。

### 3.1 `package.json`

建议修改：

- `name`
- `description`
- `author`
- `version`
- 可能不再需要的依赖

脚本部分通常可以继续保留：

- `dev`
- `build`
- `start`
- `check`
- `typecheck`
- `lint`
- `format`
- `test`

### 3.2 `.env.example`

建议根据真实项目改掉：

- 服务地址类变量
- 日志与限流配置
- Swagger 配置
- upstream 服务地址

如果真实项目会增加更多业务域服务，也应该在这里同步体现。

### 3.3 `README.md`

当前 README 偏学习型说明，迁移后建议改成真实项目版本，至少补齐：

- 项目定位
- 启动方式
- 目录结构
- 环境变量
- 核心接口
- 开发约定

### 3.4 `app.ts` 中的 core plugins 注册顺序

当前仓库已经把 core plugins 的顺序说明收口到了 `app.ts` 和项目文档中，不再单独维护 `plugins/core/README.md`。

迁移到真实项目后，建议优先修改：

- `app.ts` 中的 core plugins 注册顺序
- `README.md` 或其他项目文档里的插件职责说明
- 真实项目里是否继续保留 `docs.ts`、`under-pressure.ts` 等插件

## 四、只保留结构，不要原样照搬的部分

这一类文件的重点不是复用代码内容，而是复用“拆分方式”。

### 4.1 demo 业务模块

当前示例模块：

- `modules/live/live.routes.ts`
- `modules/live/live.schema.ts`
- `modules/live/live.controller.ts`
- `modules/live/live.service.ts`
- `modules/live/live.types.ts`

建议迁移时：

- 保留模块拆分方式
- 删除演示数据
- 删除演示字段
- 删除 `failSegments` 这类纯 demo 参数
- 改成真实业务模块名称和真实返回结构

### 4.2 route plugin 中的 demo 入口

例如：

- `plugins/routes/live.ts`

迁移后不建议继续保留 `live` 这个 demo 路由入口，而是按真实业务改成：

- `live-room.ts`
- `auth.ts`
- `wechat.ts`
- `danmaku.ts`
- `feedback.ts`

## 五、推荐的真实项目模块拆分

如果你后面要做的是直播 H5 的 BFF，比较推荐先建下面这些模块：

```text
modules/
  live-room/
    live-room.routes.ts
    live-room.schema.ts
    live-room.controller.ts
    live-room.service.ts
    live-room.types.ts
  auth/
    auth.routes.ts
    auth.schema.ts
    auth.controller.ts
    auth.service.ts
    auth.types.ts
  wechat/
    wechat.routes.ts
    wechat.schema.ts
    wechat.controller.ts
    wechat.service.ts
    wechat.types.ts
  danmaku/
  feedback/
```

推荐先从一条最核心的主链路开始，例如：

```text
GET /api/live-room/init
```

等这一条链路走通以后，再逐步补：

- 短信登录
- 微信授权
- 弹幕
- 投诉建议
- 商品和支付相关能力

## 六、建议保留的公共接口

迁移到真实项目后，下面这些接口建议保留：

- `GET /health`
- `GET /readiness`
- `GET /metrics`

原因：

- 运维和部署通常需要健康检查
- readiness 适合反映进程与平台能力状态
- metrics 方便后续接监控平台或自定义指标观测

## 七、不建议带入真实项目的内容

下面这些内容建议只保留“思路”，不要原样复制到真实业务里：

- 演示商品数据
- 演示主播数据
- `failSegments` 这种模拟降级参数
- 当前 demo 的聚合响应字段
- 学习型注释和文案
- 教学导图中的描述性文本

这些内容适合帮助理解项目结构，不适合作为真实业务接口设计依据。

## 八、最推荐的迁移顺序

建议按下面顺序执行：

1. 复制当前仓库结构。
2. 修改 `package.json`、`.env.example`、`README.md`。
3. 保留 `config / errors / lib / plugins / types / test`。
4. 删除或清空 `modules/live` 中的 demo 业务实现。
5. 按真实业务重新建立模块目录。
6. 先实现 `GET /api/live-room/init`。
7. 再逐步补 `auth / wechat / danmaku / feedback` 等模块。

## 九、一句话结论

当前这个仓库最适合迁移的，不是 demo 业务本身，而是下面这套能力：

```text
工程配置
+ Fastify 装配方式
+ core plugins 组织方式
+ route -> schema -> controller -> service -> types 的模块分层
+ 错误处理、日志、限流、监控这些平台能力
```

因此，建议你复制“工程骨架”，重写“业务模块”。
