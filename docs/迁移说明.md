# 项目结构迁移说明

这份文档只讲一件事：如果把当前仓库迁移到真实项目，哪些内容可以保留，哪些内容必须重写。

先记住一句话：

```text
复制工程骨架，重写业务模块。
```

## 1. 可以直接保留

- `app.ts`
- `server.ts`
- `config/index.ts`
- `errors/`
- `lib/`
- `types/`
- `plugins/core/`
- `plugins/routes/health.ts`
- `plugins/routes/metrics.ts`
- `test/`
- `tsconfig.json`
- `eslint.config.mjs`
- `.prettierrc.json`
- `.prettierignore`
- `.editorconfig`
- `.github/workflows/ci.yml`

这些内容主要是平台能力和工程骨架，和具体业务耦合较低。

## 2. 复制后优先修改

- `package.json`
  - 改 `name`、`description`、`author`
  - 检查依赖是否仍然需要
- `.env.example`
  - 改成真实环境变量
- `README.md`
  - 改成真实项目说明
- `app.ts`
  - 按真实项目调整 core plugins 注册顺序
- `plugins/routes/live.ts`
  - 改成真实业务模块入口

## 3. 不要原样照搬

- `modules/live/live.routes.ts`
- `modules/live/live.schema.ts`
- `modules/live/live.controller.ts`
- `modules/live/live.service.ts`
- `modules/live/live.types.ts`
- demo 返回数据
- demo 聚合字段
- `failSegments` 这类纯演示参数
- 学习型文案和注释

这些内容适合保留“拆分方式”，不适合直接带入真实业务。

## 4. 推荐的新项目模块

如果以后做直播 H5 BFF，可以优先考虑：

- `live-room`
- `auth`
- `wechat`
- `danmaku`
- `feedback`

建议先打通一条最关键的链路，例如：

```text
GET /api/live-room/init
```

## 5. 建议保留的公共接口

- `GET /health`
- `GET /readiness`
- `GET /metrics`

因为这些接口通常和部署、监控、运维长期相关。

## 6. 推荐迁移顺序

1. 复制当前仓库结构
2. 修改 `package.json`、`.env.example`、`README.md`
3. 保留 `config / errors / lib / plugins / types / test`
4. 删除 `modules/live` 的 demo 实现
5. 新建真实业务模块
6. 先做通一条主链路，再逐步扩展

## 7. 一句话结论

```text
当前仓库最适合迁移的是工程装配、平台能力和模块分层，不是 demo 业务本身。
```
